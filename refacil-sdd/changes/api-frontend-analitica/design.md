## Patrones y convenciones detectados

- **Idioma:** español formal orientado a audiencia técnica senior (regla `Always` en `.agents/summary.md`); identificadores, rutas, nombres de endpoint, esquemas JSON, comandos y fragmentos técnicos en inglés.
- **Diagramas:** Mermaid para todo diagrama nuevo; `sequenceDiagram` para flujos de solicitud/respuesta entre servicios; `stateDiagram-v2` para el diagrama de estados del incidente; `flowchart LR` o `graph TD` para topologías de componentes; `C4Context` o equivalente Mermaid para el diagrama C4 L2 del overview.
- **Terminología canónica:** "hot path / cold path", "lista roja", "discriminador de tenant", "URL pre-firmada", "ventana deslizante", "circuit breaker", "fan-out de alertas", "hexbinning H3", "audit trail", "write-back", "multi-tenant desde el primer día"; mantener consistencia con la propuesta principal y con los documentos de `almacenamiento-lectura`.
- **Referencias cruzadas ADR:** toda decisión técnica menciona el ADR que la origina (ADR-005 hexagonal, ADR-008 CQRS pragmático, ADR-011 multi-tenant); los ADRs nuevos de este change (`adr-api-gateway.md`, `adr-realtime-protocol.md`, `adr-cartography.md`) deben indicar que complementan ADR-005 y ADR-011.
- **Hexagonal (ADR-005):** cada microservicio define sus puertos de forma explícita; nombrar los puertos de forma consistente con los del resto del sistema: `EventsIndexPort`, `EventsRepositoryPort`, `ObjectStoragePort`, `AlertsKafkaPort`, `AlertsStatePort`, `CountrySyncPort`, `AnalyticsRepositoryPort`; los documentos de especificación del servicio incluyen la definición del puerto (firma de la interfaz descriptiva, no código).
- **Esquemas de respuesta JSON:** incluir al menos un ejemplo completo de solicitud y respuesta por endpoint en los documentos de especificación del servicio y en las specs OpenAPI; los ejemplos deben usar datos realistas (placas colombianas, coordenadas de Bogotá, timestamps ISO 8601).
- **Privacidad:** los documentos deben explicitar que las thumbnails son imágenes de vehículos (no rostros ni biometría); `owner_dni_hash` en `stolen_vehicles` es hash unidireccional (no PII directa); las URLs pre-firmadas de lectura tienen TTL corto (5 min) para minimizar el riesgo de exposición; `country_code` como barrera de aislamiento entre países (ADR-011).
- **Versionado de API:** prefijo `/v1/` obligatorio en todos los endpoints; los documentos OpenAPI especifican la versión de la API y el servidor base; sin breaking changes dentro de v1.

## Nota de validación cruzada entre repositorios

Este change consume contratos de lectura de `almacenamiento-lectura` y produce contratos de notificación hacia `sincronizacion-paises`. Consultar `refacil-prereqs/BUS-CROSS-REPO.md` para el proceso de acuerdo formal:

- **`almacenamiento-lectura` (upstream, aprobado):** los contratos de lectura de los cuatro almacenes (queries OpenSearch, schemas PostgreSQL+PostGIS, vistas ClickHouse, estructura de clave Redis) documentados en `design.md` de ese change son la especificación de interfaz que los puertos hexagonales de este change deben implementar. Cualquier cambio en esos contratos requiere notificación y re-validación con este change.
- **`backbone-procesamiento` (upstream, aprobado):** el schema del payload del tópico `vehicle.alerts` consumido por `alerts-service` debe ser acordado formalmente; el campo `zone` en el mensaje de alerta es crítico para el enrutamiento por zona. Cualquier cambio en el schema del mensaje requiere notificación y re-validación.
- **`sincronizacion-paises` (upstream, aprobado):** la interfaz del `CountrySyncPort` que `incident-service` usa para notificar cierres de caso (API, payload, comportamiento ante write-back no soportado) debe acordarse formalmente con ese change. Si ese change no define una interfaz de entrada para notificaciones de cierre, la coordinación es prerrequisito de implementación de `incident-service`.
- **`identidad-seguridad` (downstream, pendiente):** el schema del JWT (nombres exactos de claims: `country_code`, `role`, `zone`), la URL del JWKS endpoint y la estrategia de caché de claves públicas son el contrato entre Keycloak y el API Gateway de este change. Cualquier cambio en los nombres de claims o en la estructura del token requiere re-validación formal con este change antes de la implementación de `identidad-seguridad`.
