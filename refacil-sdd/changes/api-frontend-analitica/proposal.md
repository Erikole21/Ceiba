# proposal: api-frontend-analitica

## Objetivo

Los datos del Sistema Anti-Hurto de Vehículos ya están estructurados y almacenados en `almacenamiento-lectura` (PostgreSQL+PostGIS, OpenSearch, ClickHouse, Redis), pero tres perfiles de usuario necesitan acceder a ellos a través de interfaces adaptadas a sus flujos de trabajo específicos: el oficial de policía que busca placas, recibe alertas en tiempo real y gestiona el ciclo de recuperación de vehículos; el analista que explora mapas de calor y tendencias por ciudad y país; y el administrador que gestiona dispositivos y la lista canónica de vehículos hurtados. Este change define la capa de API, frontend y analítica que expone esos datos mediante microservicios de dominio orquestados por un API Gateway centralizado, una Web App React multi-tenant y una solución BI embebida con Apache Superset — completando el Pilar 6 (Hot path / Cold path separados) de la arquitectura propuesta.

## Alcance

**Incluye:**
- **API Gateway**: punto de entrada único (Kong o Envoy; NGINX Ingress como alternativa cloud-agnostic); validación JWT contra el JWKS endpoint de Keycloak; extracción de claims `country_code`, `role` y `zone`; rate-limiting con ventana deslizante por tenant+endpoint con contadores en Redis; enrutamiento a microservicios de dominio; terminación TLS, HSTS y política CORS; registro de solicitudes y respuestas para auditoría; adaptador hexagonal para API Gateway gestionado en nube.
- **search-service**: orquestación de la búsqueda de placa (OpenSearch con fuzziness AUTO → event_ids → PostgreSQL+PostGIS con coordenadas y geometría → URLs pre-firmadas de object storage con TTL 5 min para miniaturas); endpoint REST `GET /v1/plates/{plate}/events?from=&to=&country=&page=&pageSize=`; respuesta paginada con timestamp, lat/lon, device_id, confidence, thumbnail_url, country y city; SLO p95 < 300 ms en búsqueda de placa y p95 < 800 ms en render de mapa con hasta 10 K eventos; aislamiento multi-tenant por `country_code` del JWT.
- **alerts-service / WebSocket Gateway**: consumo del tópico Kafka `vehicle.alerts` (producido por `backbone-procesamiento`); enrutamiento de alertas a oficiales conectados en la misma `zone` (claim JWT); conexiones persistentes WebSocket (Socket.IO o WS nativo); endpoint REST de polling de alertas perdidas `GET /v1/alerts?since=&country=`; payload de alerta con placa, timestamp, coordenadas, thumbnail_url, device_id y alert_id.
- **incident-service**: gestión del ciclo de vida completo de recuperación de vehículo (`POST /v1/incidents/{plate}/recovery-action` — despachar unidad, escalar, cerrar caso; `GET /v1/incidents/{plate}` — estado actual y audit trail; `PATCH /v1/incidents/{id}/status` — actualizar estado); notificación a `sincronizacion-paises` al cerrar un caso para propagar el estado a la BD policial del país (si el adaptador soporta write-back); persistencia de estado y audit log en PostgreSQL (tablas `incidents` y `audit_log` de `almacenamiento-lectura`); emisión de eventos de ciclo de vida al tópico Kafka `incident.events`.
- **analytics-service**: backend para Superset y geo-analítica con tres endpoints — `GET /v1/analytics/heatmap?country=&from=&to=&resolution=8` (datos H3 de hexbinning desde vista materializada de ClickHouse), `GET /v1/analytics/trends?country=&city=&period=7d|30d|90d` (tendencias de hurto desde ClickHouse), `GET /v1/analytics/routes?country=&from=&to=` (rutas frecuentes de escape); conexión directa a ClickHouse vía adaptador hexagonal; SLO p95 < 3 s para agregación de 7 días usando vistas materializadas H3; ClickHouse no se expone directamente al frontend.
- **devices-service** (rol admin): CRUD de dispositivos, disparo de actualización OTA y monitoreo de salud; endpoints `GET /v1/devices`, `POST /v1/devices`, `POST /v1/devices/{id}/ota`.
- **vehicles-service** (rol admin): gestión de la lista canónica de vehículos hurtados con lectura y escritura en la tabla `stolen_vehicles`; endpoints `GET /v1/vehicles`, `POST /v1/vehicles`, `PATCH /v1/vehicles/{id}`.
- **Web App (React + TypeScript)**: vista de búsqueda de placa (mapa MapLibre GL con trayectoria de avistamientos, lista lateral paginada, popup con fecha/hora, ubicación y miniatura); vista de alertas en tiempo real con tarjetas de alerta por WebSocket, resaltado en mapa y botones de acción (despachar unidad, abrir incidente); panel de gestión de incidentes (estado, audit trail, botones de despacho/escalado/cierre); vista analítica (capa de mapa de calor con Kepler.gl sobre MapLibre, gráficos de tendencias, filtros de ventana temporal y país); embed o enlace a dashboards de Superset para el rol analista; multi-tenancy en la UI (el usuario solo ve datos de su `country_code`); i18n desde el primer día (es/en).
- **Apache Superset**: OSS, contenedorizado en Kubernetes; conectado a ClickHouse para dashboards BI del rol analista; integración con la Web App por embed o enlace autenticado.
- **Documentación de decisiones de diseño**: ADR sobre elección de API Gateway (Kong vs. Envoy vs. NGINX Ingress), ADR sobre protocolo de alertas en tiempo real (WebSocket vs. Server-Sent Events), ADR sobre librería cartográfica (MapLibre GL vs. alternativas); especificación OpenAPI por microservicio.
- **Documentación operativa**: Helm charts para cada microservicio y para el API Gateway (cloud y on-prem con values distintos); guía de despliegue de Superset en Kubernetes; runbooks de escalado y monitoreo; especificación de métricas Prometheus y trazas OpenTelemetry por servicio.

**Excluye:**
- La capa de almacenamiento que este change lee (`almacenamiento-lectura`, aprobado): los schemas, índices y contratos de lectura ya están definidos allí; este change implementa únicamente los adaptadores de lectura sobre esos contratos.
- El pipeline de procesamiento y la producción del tópico `vehicle.alerts` (`backbone-procesamiento`, aprobado): alerts-service es consumidor de ese tópico, no su productor.
- La sincronización de vehículos hurtados desde BDs policiales (`sincronizacion-paises`, aprobado): incident-service notifica a ese change en el cierre de casos, pero no reimplementa su lógica de sincronización ni sus adaptadores de país.
- La emisión de JWT y la configuración de realms en Keycloak (`identidad-seguridad`, pendiente): el API Gateway valida JWT ya emitidos; la configuración del realm y de los claims es responsabilidad del change separado.
- El agente de borde, la ingestión MQTT y el backbone de procesamiento: componentes upstream fuera del alcance de este change.
- Paneles de administración de Keycloak: la gestión de usuarios, roles y realms no es parte del frontend de este change.
- Pipelines de ML o detección de patrones avanzados: analytics-service expone analítica descriptiva (mapas de calor, tendencias, rutas de escape), no modelos predictivos.
- Generación de URLs pre-firmadas para upload de imágenes desde el borde: ese flujo pertenece a `ingestion-mqtt`; este change solo genera URLs pre-firmadas de lectura (GET) para miniaturas ya almacenadas.

## Justificación

Con `almacenamiento-lectura` aprobado, los datos están disponibles pero inaccesibles para los usuarios finales. Sin la capa de API y frontend, el pipeline completo carece de interfaz operacional: los oficiales no pueden buscar placas, recibir alertas ni gestionar incidentes; los analistas no pueden explorar tendencias ni zonas de hurto; los administradores no pueden gestionar dispositivos ni la lista de vehículos hurtados. Este change cierra el ciclo funcional end-to-end del sistema, completando el Pilar 6 de la arquitectura propuesta y habilitando la validación funcional completa de la propuesta edge-to-cloud. La separación en microservicios de dominio independientes (search, alerts, incidents, analytics, devices, vehicles) permite escalarlos de forma diferenciada según la carga real: el search-service y el WebSocket Gateway tienen patrones de tráfico muy distintos al analytics-service, justificando su desacoplamiento desde el diseño inicial.

## Restricciones

- **SLO hot path — búsqueda operacional**: search-service p95 < 300 ms para búsqueda de placa; p95 < 800 ms para render de mapa con hasta 10 K eventos simultáneos.
- **SLO hot path — alertas en tiempo real**: tiempo de propagación de alerta desde Kafka hasta el oficial conectado p95 < 2 s (end-to-end, incluyendo el WebSocket Gateway).
- **SLO cold path**: analytics-service p95 < 3 s para agregación de 7 días sobre vistas materializadas H3 de ClickHouse.
- **Cloud-agnostic**: todos los microservicios corren en Kubernetes; API Gateway con adaptador hexagonal para alternativas gestionadas en nube; mismo Helm chart para cloud y on-prem mediante values distintos; sin dependencias propietarias de nube en el código de los servicios.
- **Hexagonal (ADR-005)**: puertos/adaptadores para object storage (generación de URLs pre-firmadas de lectura), consumidor Kafka (alertas en tiempo real), consultas ClickHouse (analítica), consultas PostgreSQL y OpenSearch (búsqueda), notificación a sincronizacion-paises (incident write-back).
- **Multi-tenant desde el primer día (ADR-011)**: `country_code` del JWT como discriminador de tenant en todos los microservicios; ningún dato de un país es visible desde el contexto de otro; la UI no muestra datos de países diferentes al del usuario autenticado.
- **Autenticación y autorización**: JWT emitido por Keycloak validado en el API Gateway contra el JWKS endpoint; claims `country_code`, `role` y `zone` propagados como headers internos a los microservicios de dominio; los microservicios confían en el Gateway y no validan JWT directamente.
- **Versionado de API**: prefijo `/v1/` en todos los endpoints; especificación OpenAPI por microservicio publicada en el mismo repositorio de documentación; sin breaking changes dentro de la versión v1.
- **Observabilidad**: logs estructurados JSON, métricas Prometheus y trazas OpenTelemetry en todos los microservicios; aplicaciones 12-factor.
- **Tecnología de microservicios**: NestJS o Go; la elección por servicio se documenta en el ADR correspondiente; no se mezclan ambos en un mismo servicio.
- **Identidad pendiente**: hasta que `identidad-seguridad` esté aprobado, el API Gateway asume que Keycloak emite JWT con claims `country_code`, `role` y `zone`; cualquier cambio en el schema del JWT requerirá re-validación formal con ese change (ver `refacil-prereqs/BUS-CROSS-REPO.md`).
